2026-06-18 repo: Phantom-OS-KubernetesOptions branch: feat/wm-inference-deployment stack: core ns: positronic precedent: phantom-sonic (PR #67)

1. Goal & context

The World-Model repo (imu-policy/world-model) ships an on-robot C++ inference service in inference/ with a standalone manifest at inference/deploy/daemonset-nvidia.yaml. It reads a camera frame, encodes with V-JEPA, applies PCA, runs a 2-step-ODE flow-matching predictor, and serves a goal latent z_ref to an action-solver over POSIX shm rings (/wm_zref_req ↔ /wm_zref_resp).

This design folds that standalone manifest into the fleet's GitOps paradigm — the same shape used by positronic-control, phantom-locomotion, and most recently phantom-sonic: a base manifest under the core stack, a foundation.bot/has-* node-label gate, image-override + per-host config plumbing in host-config.py, a bootstrap config phase, and convenience commands in positronic.sh.

Closest precedent = phantom-sonic. Sonic is the most recent example of adding a new mutually-exclusive, multi-container "brain" Pod. This design mirrors its file set and wiring almost 1:1. Where it deviates (config delivery), §6 explains why.

2. Architecture

The unit of deployment is a DaemonSet Pod that is the robot's "brain." Today it runs one active container, wm-inference. It reserves a co-located action-solver container slot (Phase 2 — no shipping image yet, only the bench ping_client). The action-solver is the shm-ring producer/client (it creates /wm_zref_req); when its image exists it drops into the same Pod and shares /dev/shm. Because that Pod (via its action-solver) ultimately drives the robot, it is mutually exclusive with the other brains.

4. Mutual exclusion (extend the existing N-way check)

The validator already generalized to an enabled_drivers list when sonic landed (host-config.py ≈ L2749). The brain group is {has-positronic (default-on), has-locomotion, has-sonic}; this adds has-wm-inference as the fourth member. Rule unchanged: at most one may be "true". Since positronic defaults on, enabling wm-inference requires explicitly setting has-positronic: "false".

effective_wm = nl.get("foundation.bot/has-wm-inference", "false")
enabled_drivers = [label for label, eff in (
    ("foundation.bot/has-positronic", effective_pos),
    ("foundation.bot/has-locomotion", effective_loc),
    ("foundation.bot/has-sonic",      effective_sonic),
    ("foundation.bot/has-wm-inference", effective_wm),   # NEW
) if eff == "true"]
# len(enabled_drivers) > 1  -> same error messaging, now naming 4 labels
Semantic note: wm-inference alone only serves z_ref — it doesn't drive joints. It joins the brain group because its co-located action-solver does. Treating the Pod as the exclusion unit keeps "one brain per robot" honest even while the action-solver slot is empty (the slot is reserved, not the workload).

6. Per-host config & delivery

The worldModel: host-config block

worldModel:
  cameraId: 0                       # -> WM_CAMERA_ID  (0..5 board/eye index)
  baseSha: 33cc3ad3ab7cfc92f8eb…    # -> WM_BASE_SHA   (sha1(base ckpt)[:40])
  freshnessMs: 200                  # -> WM_FRESHNESS_MS (optional)
  tasks:                            # id <-> engine pairs (kept paired to
    - {id: honda_reach_insert, engine: honda_reach_insert.engine}   # prevent
    - {id: honda_rehome,       engine: honda_rehome.engine}         # mismatch
# derived ->  WM_TASK_IDS          = honda_reach_insert,honda_rehome
#             WM_PREDICTOR_ENGINES = /wm-models/honda_reach_insert.engine,
#                                    /wm-models/honda_rehome.engine

Scalars (cameraId, baseSha, freshnessMs) follow the flat DEFAULT_*+FIELD_TO_ENV pattern from DEFAULT_SONIC. The tasks list is the one enrichment over sonic's pure-scalar emitter: it derives the two parallel CSV vars (WM_TASK_IDS / WM_PREDICTOR_ENGINES) — matching exactly how wm_inference_main's loadConfigFromEnv() consumes them. A bare/absent worldModel block renders a complete, working default ConfigMap.

Delivery: sonic-style host-local ConfigMap — a deliberate change from the brainstorm summary's "positronic-style".

During brainstorming I floated positronic-style drift-safe delivery (base CM in git + strategic-merge into the Argo app). On reading the code: positronic needs that only because its in-Argo PHANTOM_CMD merge is bespoke, and positronic-control runs an image entrypoint with no shell-defaults. wm-inference can do better, simpler: wm_inference_main reads pure WM_* env, so wrapping it in /bin/sh -c 'export WM_X=${WM_X:-default}; exec …' gives first-boot defaults, and envFrom (optional) supplies per-host overrides once bootstrap applies the host-local CM. This is exactly the phantom-sonic mechanism — the precedent you named — with no base CM (so no ArgoCD conflict) and no bootstrap surgery. If you'd rather keep strict positronic-style drift-safety, it's a localized swap of the bootstrap phase + a base configmap.yaml.

Bootstrap wm-config phase (mirrors sonic_config)

  • New flag --wm-config; SKIP_WM_CONFIG; SELECTED_PHASES case entry; called after sonic_config in the run list.
  • Renders /etc/phantomos/wm-inference-config.yaml (ConfigMap wm-inference-config, ns positronic) from host-config.py … get-world-model-config-kv, kubectl apply -f, then rollout restart ds/wm-inference if present.
  • Host-resident, ArgoCD-unmanaged, --reset-preserved — identical lifecycle to phantom-sonic-config.

Manifest env wiring

command: ["/bin/sh","-c"]
args:
  - |
    export WM_CAMERA_ID="${WM_CAMERA_ID:-0}"
    export WM_BASE_SHA="${WM_BASE_SHA:-33cc3ad3ab7cfc92f8eb4cf177ba552680e2def6}"
    export WM_TASK_IDS="${WM_TASK_IDS:-honda_reach_insert,honda_rehome}"
    export WM_PREDICTOR_ENGINES="${WM_PREDICTOR_ENGINES:-/wm-models/honda_reach_insert.engine,/wm-models/honda_rehome.engine}"
    export WM_FRESHNESS_MS="${WM_FRESHNESS_MS:-200}"
    exec /opt/wm-inference/build/wm_inference_main
envFrom:
  - configMapRef: { name: wm-inference-config, optional: true }   # per-host overrides
env:    # static image-contract paths stay inline (NOT in the CM)
  - { name: WM_VJEPA_ENGINE,   value: /wm-models/vjepa.engine }
  - { name: WM_SIGLIP_ENGINE,  value: /wm-models/siglip_text.engine }
  - { name: WM_PCA_PATH,       value: /wm-models/pca_basisA.bin }
  - { name: WM_SPM_PATH,       value: /wm-models/spiece.model }
  - { name: WM_REGISTRY,       value: /wm-models/registry.json }
  - { name: WM_READY_FILE,     value: /tmp/wm_ready }
  - { name: WM_METRICS_PATH,   value: /tmp/wm_metrics.json }
  - { name: NVIDIA_VISIBLE_DEVICES, value: all }
  - { name: NVIDIA_DRIVER_CAPABILITIES, value: all }
Precedence is correct: when the CM is present, envFrom sets WM_*, and the ${WM_X:-default} wrapper is a no-op (the var is already set). When the CM is absent (first boot), the wrapper supplies the defaults. The static engine-path env: entries are not in the CM, so no precedence conflict arises.

9. Assumptions & open points

  • CI publishes foundationbot/wm-inference + foundationbot/wm-inference-models (arm64/Thor engines). The base manifest only references local-registry placeholders.
  • wm_inference_main lives at /opt/wm-inference/build/wm_inference_main (confirmed in Dockerfile.thor); the shell-default wrapper execs it. If a future image changes the path, update the wrapper.
  • Action-solver: documented slot only; ships inert until a real image/command exists. Pod remains the exclusion unit.
  • No /recordings mount (WM writes metrics to /tmp); add via a future DEPLOYMENT_TARGETS entry if needed.
  • Config delivery is sonic-style host-local CM (see §6 callout) rather than positronic-style — flagged for confirmation.
